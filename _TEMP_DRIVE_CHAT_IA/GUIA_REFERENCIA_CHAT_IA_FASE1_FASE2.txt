GUIA DE REFERENCIA OPERATIVA - CHAT IA SMARTHOME
Version: 1.0
Fecha: 2026-04-04

OBJETIVO DE ESTA GUIA
Esta guia explica, en forma practica, que hace cada archivo/documento del sistema de Chat IA, cuando debe modificarse, quien deberia modificarlo y que impacto tiene cada cambio.

Alcance de esta guia:
- Fase 1 (Definicion de archivos)
- Fase 2 (Estructura de datos y validaciones)

No incluye aun:
- Implementacion final del endpoint (Fase 3)
- Frontend del widget (Fase 4)
- Pruebas de punta a punta (Fase 5)


PRINCIPIO CLAVE DE ORDEN Y SEGURIDAD
Separar contenido por funcion evita errores y facilita mantenimiento:
1) Instrucciones de comportamiento (como actua el bot)
2) Conocimiento comercial (que sabe el bot)
3) Configuracion operativa (umbrales, alertas, limites)
4) Eventos de negocio (que acciones se disparan)
5) Estructura de datos y validaciones (que datos aceptar y guardar)

Esta separacion permite:
- Ajustar ventas sin tocar codigo
- Cambiar conocimiento sin tocar reglas de seguridad
- Auditar cambios con rapidez
- Reducir riesgo de romper flujos

QUE HACE CADA DOCUMENTO Y CUANDO EDITARLO

1) BOT - Instrucciones y Limites
Tipo: Google Docs
Funcion:
- Define el rol del bot, tono, objetivo comercial y limites de seguridad.
- Define como pedir datos y como cerrar una conversacion hacia cotizacion/asesor.

Cuando modificar:
- Si quieres cambiar el estilo de respuesta (mas breve, mas comercial, mas formal).
- Si cambian politicas de atencion.
- Si agregas o quitas datos minimos requeridos al cliente.

No modificar aqui:
- Precios, planes detallados o contenido factual extenso (eso va en Conocimiento Comercial).
- Umbrales numericos de score (eso va en Configuracion).

Impacto del cambio:
- Cambia la conducta del bot en todas las conversaciones.

Responsable sugerido:
- Lider comercial + responsable de operaciones.


2) BOT - Conocimiento Comercial
Tipo: Google Docs
Funcion:
- Contiene hechos del negocio: kits, planes, FAQ, alcances y restricciones comerciales.

Cuando modificar:
- Si cambia un plan, kit, cobertura, beneficios, politicas o FAQ.
- Si agregas nueva oferta comercial.

No modificar aqui:
- Reglas de seguridad/comportamiento del bot.
- Umbrales y puntuaciones.

Impacto del cambio:
- Cambia lo que el bot considera "verdad comercial" para responder.

Responsable sugerido:
- Equipo comercial/producto.

Recomendacion:
- Escribir informacion clara, concreta y sin contradicciones.


3) BOT - Eventos Comerciales
Tipo: Google Docs
Funcion:
- Define los eventos del negocio y sus acciones asociadas.
Ejemplos: lead_caliente, cotizacion_solicitada, cierre_probable, cuota_ia_agotada.

Cuando modificar:
- Si cambias criterios de disparo de alertas.
- Si agregas nuevos eventos o acciones.

No modificar aqui:
- Estructura tecnica de columnas de logs.

Impacto del cambio:
- Cambia que acciones ejecuta el sistema frente a cada situacion.

Responsable sugerido:
- Operaciones + comercial.


4) BOT - Configuracion
Tipo: Google Sheets
Funcion:
- Centraliza parametros numericos y booleanos:
  - umbral lead caliente
  - ponderaciones de score
  - email de alertas
  - fallback
  - limites de validacion
  - cooldown de alertas

Cuando modificar:
- Si quieres afinar conversion (subir/bajar umbral).
- Si quieres ajustar sensibilidad del scoring.
- Si cambias correo de alertas o politicas de fallback.

No modificar aqui:
- Texto largo de conocimiento comercial.

Impacto del cambio:
- Cambia el comportamiento operativo sin tocar codigo.

Responsable sugerido:
- Operaciones con validacion comercial.

Buenas practicas:
- No borrar claves existentes.
- Cambiar solo la columna valor.
- Mantener columna comentario actualizada.


5) BOT - Formato Resumen Conversacion
Tipo: Google Docs
Funcion:
- Define plantilla estandar de resumen para seguimiento comercial.
- Asegura que todas las conversaciones queden resumidas del mismo modo.

Cuando modificar:
- Si necesitas mas contexto para vendedores.
- Si se detecta que faltan datos clave en seguimiento.

No modificar aqui:
- Reglas de validacion tecnica.

Impacto del cambio:
- Mejora o empeora calidad de handoff comercial.

Responsable sugerido:
- Ventas/seguimiento.


6) BOT - Reglas Validacion
Tipo: Google Docs
Funcion:
- Define reglas minimas para aceptar datos y enviar leads.
- Estandariza validaciones de telefono, email, consentimiento y score.

Cuando modificar:
- Si cambia criterio legal/comercial de consentimiento.
- Si necesitas validacion mas estricta o flexible.

No modificar aqui:
- Detalle de productos o FAQ.

Impacto del cambio:
- Afecta calidad de datos y volumen de leads derivados.

Responsable sugerido:
- Operaciones + compliance comercial.


7) Smarthome Chat IA - Operacion
Tipo: Google Sheets
Funcion:
- Almacena la operacion diaria del chat en pestañas:
  - config
  - chat_logs
  - eventos_chat
  - errores_chat
  - leads_precalificados
  - schema_lead (si se carga Fase 2)

Cuando modificar:
- Estructura de pestañas: solo con control.
- Filas: uso diario automatizado.

No modificar aqui:
- Nombres de columnas sin coordinar, porque puede romper scripts.

Impacto del cambio:
- Puede afectar registro de datos y trazabilidad.

Responsable sugerido:
- Operaciones tecnicas.

REGLA PRACTICA: QUE EDITAR SEGUN NECESIDAD

Caso A: "Quiero que el bot sea mas vendedor"
Editar: BOT - Instrucciones y Limites

Caso B: "Cambio un plan o agrego un kit"
Editar: BOT - Conocimiento Comercial

Caso C: "Llegan muchos falsos positivos de lead caliente"
Editar: BOT - Configuracion (umbral y ponderaciones)

Caso D: "Quiero nueva alerta para oportunidades urgentes"
Editar: BOT - Eventos Comerciales

Caso E: "Los vendedores necesitan resumen mas claro"
Editar: BOT - Formato Resumen Conversacion

Caso F: "Estamos aceptando telefonos invalidos"
Editar: BOT - Reglas Validacion


CONTROL DE CAMBIOS (RECOMENDADO)
Aplicar siempre este mini protocolo:
1) Registrar que se cambio (documento + fecha + motivo)
2) Cambiar solo una cosa a la vez
3) Probar 3 conversaciones reales de ejemplo
4) Confirmar que logs y eventos siguen correctos
5) Si empeora resultados, volver al valor anterior


RIESGOS COMUNES Y COMO EVITARLOS

Riesgo 1: Mezclar conocimiento con reglas de comportamiento
- Mitigacion: mantener separacion estricta entre Instrucciones y Conocimiento.

Riesgo 2: Romper parsing por cambios de estructura
- Mitigacion: no cambiar encabezados/columnas sin actualizar codigo.

Riesgo 3: Alertas excesivas por cuota agotada
- Mitigacion: respetar cooldown de alertas en configuracion.

Riesgo 4: Datos de lead incompletos
- Mitigacion: exigir consentimiento y contacto minimo valido.

Riesgo 5: Derivar demasiados leads de baja calidad
- Mitigacion: ajustar umbral y ponderaciones gradualmente.
